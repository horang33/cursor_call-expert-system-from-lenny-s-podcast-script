# 기여 가이드

Lenny's Podcast 전문가 조언 시스템에 기여해주셔서 감사합니다! 🎉

## 🤝 기여 방법

### 1. 버그 리포트

버그를 발견하셨나요?

1. GitHub Issues에서 기존 이슈 검색
2. 없다면 새 이슈 생성
3. 다음 정보 포함:
   - 문제 설명
   - 재현 단계
   - 예상 동작
   - 실제 동작
   - 환경 정보 (OS, Cursor 버전)

### 2. 기능 제안

새로운 기능 아이디어가 있나요?

1. GitHub Issues에서 "Feature Request" 생성
2. 다음 내용 포함:
   - 기능 설명
   - 사용 사례
   - 예상 효과
   - 대안 고려사항

### 3. 문서 개선

문서를 개선하고 싶으신가요?

- 오타 수정
- 설명 개선
- 예시 추가
- 번역 개선

### 4. 코드 기여

코드를 개선하고 싶으신가요?

1. Fork 저장소
2. 새 브랜치 생성
3. 변경사항 작성
4. Pull Request 제출

## 🔧 개발 환경 설정

### 필요한 것

- Git
- 텍스트 에디터
- Cursor IDE (테스트용)
- Markdown 지식

### 저장소 클론

```bash
git clone [REPO_URL]
cd lennys-expert-advisor
```

### 브랜치 전략

- `main`: 안정 버전
- `develop`: 개발 버전
- `feature/기능명`: 새 기능
- `fix/버그명`: 버그 수정
- `docs/문서명`: 문서 개선

### 브랜치 생성

```bash
git checkout -b feature/새로운-전문가-추가
```

## 📋 Pull Request 가이드

### PR 체크리스트

- [ ] 변경사항이 명확히 설명됨
- [ ] 관련 이슈가 링크됨
- [ ] 문서가 업데이트됨 (필요시)
- [ ] 예시가 추가됨 (필요시)
- [ ] 테스트 완료
- [ ] 커밋 메시지가 명확함

### PR 템플릿

```markdown
## 변경사항
- 무엇을 변경했는지 설명

## 동기
- 왜 이 변경이 필요한지 설명

## 테스트
- 어떻게 테스트했는지 설명

## 스크린샷 (있다면)
- 변경사항 시각화

## 관련 이슈
- Closes #이슈번호
```

### 커밋 메시지 규칙

```
타입: 간단한 설명

상세 설명 (선택)

관련 이슈: #123
```

**타입:**
- `feat`: 새 기능
- `fix`: 버그 수정
- `docs`: 문서 변경
- `style`: 코드 포맷팅
- `refactor`: 리팩토링
- `test`: 테스트 추가
- `chore`: 기타 변경

**예시:**
```
feat: Design 전문가 추가

- design.md 파일 생성
- .cursorrules에 Design Expert 추가 (키워드 트리거 포함)
- README에 Design 전문가 설명 추가

관련 이슈: #45
```

## 🎨 스타일 가이드

### Markdown 스타일

- 제목: `#`, `##`, `###` 사용
- 리스트: `-` 또는 `1.` 사용
- 코드 블록: ` ``` ` 사용
- 링크: `[텍스트](URL)` 형식

### 한국어 작성 규칙

- 존댓말 사용 ("~합니다", "~하세요")
- 명확하고 간결한 문장
- 전문 용어는 한글(영어) 병기
- 예: "제품-시장 적합성(Product-Market Fit, PMF)"

### 코드 스타일

- 들여쓰기: 2 spaces
- 줄 길이: 100자 이하 권장
- 주석: 한국어로 작성

## 🌟 좋은 기여 예시

### 예시 1: 새로운 전문가 추가

```markdown
# Design 전문가

## 개요
UI/UX 디자인, 디자인 시스템, 사용자 경험 분야의 전문가입니다.

## 대표 게스트

- ...

## 관련 토픽
- Design (1개 에피소드)

## 이 전문가에게 물어보면 좋은 질문
- "디자인 시스템을 어떻게 구축하나요?"
- ...
```
